SECCIÓN IV – PREGUNTAS EXPOSITIVAS (4 preguntas) 5% C/U Total 20%
Instrucciones: Responde de forma desarrollada y fundamentada. Extensión sugerida: 1 a 2 párrafos por pregunta.
1. Explica en detalle el principio de Inversión de Dependencias (DIP) de SOLID. Incluye un ejemplo concreto de cómo se aplica en código TypeScript y argumenta por qué es fundamental para la mantenibilidad del software.
Respuesta desarrollada:

El principio de Inversión de Dependencias (DIP) establece dos reglas fundamentales: los módulos de alto nivel (la lógica de negocio central) no deben depender de módulos de bajo nivel (detalles de infraestructura como bases de datos o servicios externos), sino que ambos deben depender de abstracciones (interfaces). Asimismo, las abstracciones no deben depender de los detalles, sino los detalles de las abstracciones.

DIP es fundamental para la mantenibilidad porque desacopla el software. Permite modificar o reemplazar componentes tecnológicos enteros sin alterar ni una sola línea de código de la lógica de negocio, reduciendo drásticamente la regresión de errores.

// ABSTRACCIÓN (Interface) interface UserRepository { save(user: any): void; } // DETALLE / BAJO NIVEL (Implementación concreta) class MySQLUserRepository implements UserRepository { save(user: any) { console.log("Guardando usuario en MySQL..."); } } // ALTO NIVEL (Lógica de negocio orientada a la abstracción) class UserService { private userRepository: UserRepository; constructor(userRepository: UserRepository) { this.userRepository = userRepository; } createUser(user: any) { this.userRepository.save(user); } }
2. Describe el proceso completo de Ingeniería de Requisitos, desde la elicitación hasta la validación. Explica cómo la IA puede asistir en cada fase y menciona al menos dos técnicas de elicitación con ejemplos concretos.
Respuesta desarrollada:

La Ingeniería de Requisitos es un ciclo de vida estructurado que comprende cuatro fases principales: 1) Elicitación (descubrimiento y recopilación de necesidades mediante técnicas con stakeholders); 2) Análisis y Negociación (estudio de la viabilidad y priorización); 3) Especificación (documentación formal de requisitos mediante SRS o Historias de Usuario); y 4) Validación (revisión conjunta con el cliente para asegurar que el documento refleja fielmente sus necesidades reales).

La Inteligencia Artificial puede potenciar cada fase de manera sustancial. En la elicitación, un LLM puede sugerir preguntas clave. En el análisis, identifica inconsistencias lógicas. En la especificación, transforma notas informales en historias estructuradas. Finalmente, ayuda a generar listas de verificación o casos de prueba automáticos basados en la documentación.

Ejemplos de técnicas de elicitación:
• Entrevistas estructuradas: Reunirse con el gerente financiero para preguntarle qué reportes fiscales obligatorios debe generar el sistema.
• Observación directa (Job Shadowing): Sentarse junto a un cajero para registrar cuántas pulsaciones le toma procesar una devolución.

3. Explica la diferencia entre los diagramas de Clases, Secuencia y Estados en UML. Para cada uno, describe su propósito, qué elementos lo componen y proporciona un ejemplo de cuándo utilizarías cada uno en el desarrollo de un sistema de e-commerce.
Respuesta desarrollada:

La diferencia principal radica en la perspectiva del sistema que capturan: el diagrama de clases es estático, mientras que los de secuencia y estados son de tipo dinámico.

  • Diagrama de Clases: Modela la estructura estática. Compuesto por clases (atributos y métodos) y sus relaciones. En un e-commerce se usa para modelar las entidades Producto y Carrito.
  • Diagrama de Secuencia: Ilustra el intercambio de mensajes entre objetos ordenados cronológicamente. Compuesto por líneas de vida, activaciones y mensajes. Se utiliza para detallar el flujo exacto del proceso de pago invocando a la API de pasarela de pago.
  • Diagrama de Estados: Representa el ciclo de vida de un objeto reactivo individual en respuesta a eventos. Compuesto por estados, transiciones y acciones. Se aplica para modelar el flujo de un Pedido ('Pendiente', 'Pagado', 'Enviado', 'Entregado').
4. Analiza el siguiente escenario y responde:

"Una startup está desarrollando un sistema de gestión de salud digital. El equipo es pequeño (5 desarrolladores) y los requisitos cambian frecuentemente por regulaciones y feedback de usuarios. Tienen 3 meses para tener un MVP (Producto Mínimo Viable)."

Basado en los conceptos vistos en clase, recomienda:
a) ¿Qué modelo de ciclo de vida utilizarías y por qué?
b) ¿Qué principios de diseño aplicarías para asegurar mantenibilidad?
c) ¿Cómo integrarías IA en el proceso de desarrollo?
d) ¿Qué diagramas UML priorizarías para la documentación?
Respuestas recomendadas al escenario:

a) Ciclo de vida: Se debe elegir Scrum con sprints de 1 o 2 semanas. Un enfoque iterativo e incremental garantiza entregar valor rápido y pivotar sin desperdiciar recursos frente a regulaciones cambiantes.

b) Principios de diseño: Se deben aplicar principios SOLID acoplados a una Clean Architecture. Esto garantizará que las normativas regulatorias de salud estén aisladas de la base de datos o de la interfaz de usuario.

c) Integración de IA: Se utilizarán asistentes de código (como GitHub Copilot) para acelerar la escritura de pruebas unitarias y código repetitivo, optimizando el desarrollo del MVP.

d) Diagramas UML prioritarios: Se priorizará un enfoque ágil documentando el Diagrama de Clases (modelo de dominio base), Historias de Usuario / Casos de Uso para delimitar el alcance del MVP, y un Diagrama de Secuencia para el flujo crítico de manejo y encriptación de datos sensibles de salud.
